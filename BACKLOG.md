# Backlog — Trustpilot Scraper & Analyzer

---

## Контекст проекта

**Что это:** инструмент для сбора, анализа и визуализации отзывов с Trustpilot для четырёх немецких платформ аренды жилья: immobilienscout24, rentumo, immosurf, immowelt.

**Стек:** Python 3.14, SQLite, OpenAI API, Playwright, LangGraph, Streamlit, gspread.

**Ключевые файлы:**
| Файл | Роль |
|------|------|
| `scraper.py` | Сбор отзывов через Playwright → SQLite |
| `analyze.py` | Категоризация отзывов через OpenAI → SQLite |
| `agent.py` | OpenAI-агент с tool use (get_reviews, get_stats, get_categories, export_to_sheets) |
| `pipeline.py` | LangGraph pipeline: проверка данных → категории → отчёт |
| `app.py` | Streamlit UI: вкладки агент / отчёт, сайдбар с экспортом |
| `sheets.py` | Экспорт данных в Google Sheets |
| `tools.py` | Определения инструментов агента + call_tool dispatcher |
| `logger.py` | Структурированные JSON-логи, ротация по дням (30 дней) |

**База данных:** `data/reviews.db`
- `reviews` — отзывы (id, company, title, text, rating, reply, published_date, ...)
- `categories` — результаты анализа LLM (company, group_type, name, description, count, analyzed_at)

**Известные ограничения:**
- Service account Google не может создавать файлы (Drive quota = 0). Пользователь сам создаёт таблицу и расшаривает сервисному аккаунту как Editor. ID таблицы хранится в `config/sheets_config.json`.
- gpt-5.2 обрывает соединение на больших payload. Решение: батчинг по 50 отзывов + 3 ретрая с паузами 5s → 10s → 20s (в `analyze.py`).
- Trustpilot по умолчанию отдаёт только англоязычные отзывы. Скрапер добавляет `?languages=all` ко всем URL через `_build_url()` в `scraper.py`.

## Беклог

---

### [ИДЕЯ] Диалоги в агенте

**Приоритет:** высокий — минимум кода, сразу меняет UX

**Что даёт:** агент помнит предыдущие вопросы в рамках сессии. Сейчас каждый вызов `run_agent` начинается с чистого листа. С диалогами становится возможным: "а покажи мне эти отзывы", "сравни с предыдущим результатом", уточняющие вопросы.

**Реализация:**

1. `agent.py` — `run_agent` принимает опциональный `history: list` и возвращает обновлённую историю:
   ```python
   def run_agent(question: str, model: str = "gpt-4o-mini",
                 history: list | None = None,
                 on_tool_call=None) -> tuple[str, list]:
       messages = history or []
       messages = [{"role": "system", "content": SYSTEM_PROMPT}] + messages
       messages.append({"role": "user", "content": question})
       # ... tool loop ...
       # возвращаем ответ + messages без system prompt (для хранения)
       return answer, messages[1:]  # срезаем system
   ```

2. `app.py` — хранить историю в `st.session_state["history"]`, передавать в `run_agent`, показывать предыдущие сообщения в UI как чат.

**Комментарии:**
- История растёт неограниченно — при длинных сессиях можно упереться в context window модели. Решение на потом: обрезать до последних N сообщений или суммаризировать.
- System prompt не нужно хранить в истории — добавляем его каждый раз при вызове.

### [ИДЕЯ] Анализ ответов компаний

**Приоритет:** средний

**Что даёт:** агент умеет анализировать как компании отвечают на отзывы — какие аргументы используют, в какой тональности пишут, есть ли шаблонные фразы. Поддерживает сравнение между компаниями: "кто отвечает эмпатичнее — immowelt или immobilienscout24?"

**Реализация:**

Живой инструмент — при вызове делает поход в LLM, результаты в БД не сохраняются.

1. `tools.py` — обновить `get_reviews`: добавить поле `reply` в возвращаемые данные (сейчас возвращается только `has_reply: bool`, текст ответа недоступен агенту).

2. `tools.py` — новый инструмент `analyze_replies(company, limit=50)`:
   ```python
   def analyze_replies(company: str, limit: int = 50) -> dict:
       # выбрать отзывы с непустым reply из БД
       # отправить тексты ответов в LLM
       # вернуть: top_arguments, tone, patterns
   ```

3. LLM-промпт возвращает структуру:
   - `tone` — общая тональность: эмпатичная / формальная / защитная / отписка
   - `top_arguments` — топ аргументов с частотой (["обратитесь в поддержку" × 12, ...])
   - `patterns` — шаблонные фразы которые повторяются

4. `TOOL_DEFINITIONS` — добавить описание инструмента чтобы агент знал когда его вызывать.

**Для сравнения компаний:** агент сам вызывает `analyze_replies` для каждой компании и синтезирует сравнение в финальном ответе — отдельного инструмента сравнения не нужно.

**Комментарии:**
- Данные уже в БД: колонка `reply` в таблице `reviews` заполнена скрапером, просто не используется.
- Лимит 50 ответов — разумный потолок чтобы не раздувать контекст LLM; для компаний с малым числом ответов вернёт всё что есть.
- Тональность имеет смысл анализировать только на ответах к негативным отзывам (rating ≤ 3) — именно там видна реальная реакция компании.

### [ИДЕЯ] RAG — семантический поиск по отзывам

**Приоритет:** средний

**Что даёт:** новый инструмент агента `search_reviews(query, company)` — находит отзывы, релевантные по смыслу, а не только по рейтингу. "Найди жалобы на медленную техподдержку" работает без точных слов в тексте.

**Вариант А — SQLite FTS5 (полнотекстовый поиск)**

- Виртуальная таблица `reviews_fts` с триггерами на insert/update в `reviews`
- Поиск через `MATCH`: `SELECT ... FROM reviews_fts WHERE reviews_fts MATCH ?`
- Новый инструмент `search_reviews(query, company, limit=10)` в `tools.py`
- Плюсы: нулевые зависимости, мгновенная индексация, работает офлайн
- Минусы: только keyword-поиск, не понимает синонимы и семантику. "Медленный ответ" не найдёт "долго ждал реакции".
- Подходит если нужно быстро и просто.

**Вариант Б — Эмбеддинги + ChromaDB (семантический поиск)** ← рекомендуется

- Генерируем эмбеддинги всех отзывов через `text-embedding-3-small` (~$0.01 на всю базу)
- Храним в ChromaDB (персистентная коллекция на диске в `data/chroma/`)
- Новый скрипт `embed.py` — разовый прогон + инкрементальное обновление при новом скрапинге
- Инструмент агента `search_reviews(query, company, top_k=10)` — cosine similarity по эмбеддингам
- Плюсы: реальная семантика, многоязычность, находит смысловые совпадения
- Минусы: зависимость от ChromaDB (`pip install chromadb`), нужен API-вызов для embed запроса

**Вариант В — LangChain + FAISS**
- LangChain `Document` + `FAISS` векторный индекс
- Больше зависимостей, сложнее настройка
- Оправдан только если проект переходит на LangChain-стек целиком. Сейчас избыточно.

**Комментарии:**
- Вариант А можно сделать за 1 час и сразу получить ценность для keyword-запросов.
- Вариант Б — правильный выбор для аналитики отзывов, где важна семантика.
- Можно сделать оба: FTS как фоллбэк когда ChromaDB не инициализирован.
- При новом скрапинге нужно автоматически добавлять новые эмбеддинги — добавить вызов `embed.py` в конец `scraper.py`.

### [ИДЕЯ] LangGraph Memory Agent (Итерация 3)

**Приоритет:** средний

**Что даёт:** агент с персистентной памятью между сессиями. Запоминает контекст прошлых вопросов, выводы, предпочтения пользователя. При следующем запуске знает что уже анализировалось.

**Реализация:**

LangGraph поддерживает `checkpointer` — механизм сохранения состояния графа. Основные варианты хранения:
- `MemorySaver` — только в RAM, не персистентный (для тестов)
- `SqliteSaver` — сохраняет в SQLite, персистентно, нулевые зависимости ← рекомендуется
- `AsyncSqliteSaver` — асинхронная версия для FastAPI/async окружений

**Структура графа:**

```
[Загрузить память] → [Понять вопрос] → [Tool calls] → [Сформировать ответ] → [Сохранить в память]
```

Узел "Загрузить память" достаёт из БД:
- Историю последних N диалогов
- Краткое саммари прошлых сессий (генерируется LLM при закрытии сессии)
- Заметки о компаниях (что уже выяснили, какие выводы сделали)

**Ключевые решения которые нужно принять:**
- Граница сессии: по времени (30 мин неактивности) или явная кнопка "новый диалог"
- Формат памяти: строки саммари или структурированный JSON
- Где хранить: отдельная таблица `agent_memory` в `reviews.db` или отдельный файл

**Связь с "Диалогами в агенте":**
- Диалоги — история текущей сессии (в RAM / session_state)
- Memory Agent — персистентность между сессиями (в SQLite)
- Можно делать независимо, Memory Agent не требует диалогов как пререквизит

**Комментарии:**
- LangGraph уже есть в проекте (`pipeline.py`), зависимость не новая
- `SqliteSaver` из `langgraph.checkpoint.sqlite` можно подключить к существующему `reviews.db`
- Текущий `agent.py` проще переписать как LangGraph граф, чем прикручивать поверх

## Готово

### ✅ Итерация 1 — Скрапер + Анализатор
Playwright-скрапер → SQLite. LLM-анализатор категорий (негативные/позитивные). CLI с флагами `--company`, `--reviews`, `--no-cache`. Добавлен `?languages=all` ко всем URL чтобы получать отзывы на всех языках.

### ✅ Итерация 2 — Google Sheets экспорт
`sheets.py` + инструмент `export_to_sheets` в агенте. Экспорт статистики, отзывов (с фильтром по рейтингу), категорий. Фильтр `min_rating`/`max_rating` проходит насквозь от агента до `get_reviews`. Service account без Drive quota — таблица создаётся руками, ID в `config/sheets_config.json`.

### ✅ Итерация 4 — Streamlit UI
Два таба: агент (Q&A + быстрые кнопки) и отчёт (LangGraph pipeline). Сайдбар со статистикой компаний и кнопкой экспорта. Кнопки без `disabled=` (убрано из-за задержки debounce), валидация на click.

### ✅ Логирование
Структурированные JSON-логи через `logger.py`. TimedRotatingFileHandler (daily, 30 дней). DEBUG/INFO только в файл, ERROR дублируется в терминал. События: `agent_start/done/error`, `tool_call/result`, `pipeline/*`, `sheets/export_*`. Документация в `docs/logging.md`.
